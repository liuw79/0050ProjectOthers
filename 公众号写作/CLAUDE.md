# 公众号写作 - 完整流程

## 9步写作流程

### Step 1: 理解需求 & 保存Brief

**目标：** 明确写作需求，保存brief以便后续查阅。

**操作步骤：**
1. 仔细阅读用户提供的写作需求
2. 如果是完整brief，直接保存；如果只是简单需求，需要补充信息
3. 创建文件：`../_briefs/项目名-商单brief.md`（如果是商单）或 `../_briefs/主题-brief.md`

**Brief必须包含：**
- 文章主题/核心信息
- 目标受众
- 预期字数
- 截稿时间
- 特殊要求（如需配图、需测试等）

---

### Step 2: 信息搜索与知识管理 ⭐

**目标：** 确保使用最新、最准确的信息。

**什么时候必做：**
- 涉及新产品、新技术
- 涉及2024-2025年的信息
- 需要业界最新动态

**搜索渠道：**
- 官方信息（产品官网、官方博客）
- 科技媒体报道（TechCrunch、The Verge等）
- 社区讨论（Reddit、Hacker News等）
- 竞品对比

**保存规范：**
- 文件路径：`../_knowledge_base/主题-YYYY-MM-DD.md`
- 必须包含：
  - 信息收集时间
  - 信息来源（带URL）
  - 关键发现
  - 下次更新建议

**示例：**
```markdown
# Claude Code - 2025-01-09

## 信息收集时间
2025年1月9日

## 信息来源
- Claude Code官方文档: https://...
- TechCrunch报道: https://...
- Hacker News讨论: https://...

## 关键发现
- 发布时间：2024年X月
- 核心特性：...
- 与竞品对比：...

## 下次更新建议
建议每月更新一次，关注新版本发布
```

---

### Step 3: 选题讨论 ⭐ 必做

**目标：** 确定文章方向，避免方向错误导致返工。

**重要原则：** 不要直接写文章！先讨论选题！

**操作步骤：**
1. 基于brief和调研结果，提供3-4个选题方向
2. 每个选题包含：
   - **标题**（吸引人的）
   - **核心角度**（从什么角度切入）
   - **工作量评估**（⭐⭐⭐⭐⭐五星评级）
   - **优势和劣势**
   - **是否需要真实测试**
3. 每个选题附带大纲（3-7个大标题 + 预计字数分配）
4. **等待用户选择**，不要自己假设

**示例：**
```markdown
## 选题方向建议

### 方向1: 教程式 - "Claude Code完全上手指南"
**核心角度**: 手把手教读者使用
**工作量**: ⭐⭐⭐⭐ (需要真实测试)
**优势**: 实用性强，容易吸引新手
**劣势**: 需要大量截图和测试

**大纲**:
1. 什么是Claude Code (300字)
2. 安装和配置 (500字)
3. 5个实战案例 (1500字)
4. 常见问题解决 (400字)
...

### 方向2: 观点式 - "为什么Claude Code是最好的写作Agent"
...
```

### Step 4: 创建协作文档（如需测试/配图）

**目标：** 明确哪些任务需要用户配合。

**什么时候创建：**
- 需要真实测试/使用产品
- 需要配图
- 需要用户提供特定信息

**文件路径：** `../_协作文档/文章主题-协作清单.md`

**协作文档必须包含：**
1. **测试任务清单**（如需测试）
   - 详细步骤
   - 统一Prompt（如需）
   - 数据记录表格
2. **配图需求清单**（如需配图）
   - 必需配图（带checkbox）
   - 可选配图（带checkbox）
3. **时间和成本预估**
4. **协作检查清单**

**示例：**
```markdown
# Claude Code教程 - 协作清单

## 测试任务

### 任务1: 测试代码生成功能
- [ ] 创建新项目
- [ ] 使用Prompt: "创建一个Todo应用"
- [ ] 记录响应时间、代码质量
- [ ] 截图保存到 `images/claude-code/`

## 配图需求

### 必需配图
- [ ] 安装界面截图
- [ ] 主界面截图
- [ ] 代码生成效果对比

### 可选配图
- [ ] Logo
- [ ] 功能演示GIF

## 预估
- 测试时间: 2-3小时
- 配图数量: 5-8张
```

### Step 5: 学习风格

**目标：** 确保文章符合用户的写作风格。

**操作步骤：**
1. 阅读 `../写作参考/风格指南.md`
2. 阅读本文件夹或 `../写作参考/代表作/` 中至少2-3篇用户的文章
3. 提取：
   - 开头方式（故事、观点、场景、数据等）
   - 结构偏好（总分总、并列式、递进式等）
   - 语言特征（口语化程度、句式长度、常用词汇）
   - 金句风格（有没有、什么类型）

### Step 5.5: 使用个人素材库 ⭐

**目标：** 用真实经历和观点替代AI腔，这是降AI味的核心。

**使用方法：**

#### 方法A: 直接搜索原始数据（推荐）
- 使用Grep工具在 `../个人素材库/` 中搜索关键词
- 找到用户的真实经历、观点、吐槽
- 示例：写高德扫街榜文章，搜索"高德|扫街榜"

#### 方法B: 查看已提炼素材
- 如果是常见主题，查看 `../个人素材库/主题索引.md`
- 打开对应的素材文件

**典型使用场景：**
- 文章开头：用真实经历引入
- 观点支撑：用真实评价增强可信度
- 案例展示：用真实项目案例
- 结尾思考：用个人洞察升华主题

**重要原则：**
- ✅ 所有素材都必须是真实的
- ✅ 改写成长文逻辑，不能直接复制粘贴
- ❌ 不能编造或夸大

---

### Step 6: 等待测试数据

**如果Step 4创建了协作文档：**
- 明确告知用户需要配合什么
- 等待用户完成测试任务
- 接收数据和配图
- 然后才开始写作

---

### Step 7: 创作初稿

**目标：** 基于真实数据和个人素材，创作文章初稿。

**写作要求：**
- 基于真实数据（不编造）
- 保持"实践+落地"调性
- 加入具体案例（数字、场景、对话）
- 自然融入个人经验和视角（来自个人素材库）

**初稿标准：**
- 可以不完美，重点是把内容写出来
- 审校环节会系统化优化
- 字数符合brief要求

---

### Step 7.5: 风格转换实验（可选）

**说明：** 这是实验性功能，不强制。

**操作：**
- 可以尝试用不同写作者的语言风格重新表达
- 参考对象：Keso、和菜头、梁宁、张小龙、PG等

**核心原则：**
- ✅ 只借鉴语言风格（句式、节奏、思维方式）
- ❌ 禁止使用原话（不引用别人的金句）
- ❌ 禁止用他人经历（"我"始终是用户本人）

---

### Step 8: 三遍审校（降AI味）⭐

**目标：** 系统化降低AI检测率至30%以下。

#### 第一遍：内容审校（逻辑、事实、结构）

**检查清单：**
- [ ] 事实准确？（数据、时间、产品名称、版本号）
- [ ] 逻辑清晰？（前后无矛盾、因果关系合理）
- [ ] 结构合理？（无跑题、重点突出）
- [ ] 无编造？（所有数据和案例都真实）
- [ ] 数据有来源？（关键数据注明出处）

#### 第二遍：风格审校（AI味降重）

**这是最关键的一遍**，目标是去掉AI味，增加人味。

**检查清单：**

**1. 删除套话**
- ❌ "在当今时代"
- ❌ "综上所述"
- ❌ "值得注意的是"
- ❌ "随着...的发展"
- ❌ "越来越多的人"

**2. 拆解AI句式**
- ❌ "不是...而是..."连续出现
- ❌ "既...又..."、"一方面...另一方面..."
- ❌ 过度使用排比

**3. 替换书面词汇**
- ❌ "显著提升" → ✅ 具体数字（"提升30%"）
- ❌ "充分利用" → ✅ "用好"
- ❌ "进行操作" → ✅ 直接用动词
- ❌ "实现了" → ✅ "做到了"

**4. 改成口语化**
- 用短句（15-25字为主）
- 用简单词汇
- 像说话一样写

**5. 加入真实细节**
- 抽象表达 → 具体数字/案例
- "很好" → "好在哪里？举个例子"

**6. 加入个人态度**
- 中立客观 → 明确观点
- "可能"、"或许" → "我觉得"、"确实"

**改写示例：**

❌ "在当今AI技术飞速发展的时代，编程工具也在不断进化..."
✅ "Claude Code出了。我用了两周，确实比Cursor好用。"

❌ "通过充分利用Claude Code的能力，可以显著提升开发效率..."
✅ "用好Claude Code，你的开发速度能快不少。我测下来，平均每个项目省3-5小时。"

#### 第三遍：细节打磨（标点、排版、节奏）

**检查清单：**
- [ ] 句子长度合适？（15-25字为主，不超过30字）
- [ ] 段落不太长？（手机屏幕3-5行）
- [ ] 标点自然？（多用句号，少用逗号连接长句）
- [ ] 节奏有变化？（快慢结合、长短结合）
- [ ] 大声朗读，感受节奏

**操作方法：**
- 找出超过30字的长句，拆短
- 找出超过5行的段落，拆分
- 删除不必要的逗号，改成句号

### Step 9: 文章配图 ⭐

**目标：** 直接完成配图，不只是写配图指南。

**操作步骤：**

**1. 分析文章，确定配图需求**
- 推荐5-8张图
- 标注哪些是必需、哪些是可选

**2. 创建图片文件夹**
- 路径：`../images/文章主题/`

**3. 获取/生成图片**
- 优先级：
  1. 公共领域作品（Wikimedia Commons）
  2. AI生成（如用户有火山引擎API等）
  3. 免费图库（Unsplash、Pexels）
  4. 截图/官方素材（需注明来源）

**4. 在Markdown中插入图片**
- 使用绝对路径：`![描述](/Users/.../images/文章主题/图片名.jpg)`
- 或相对路径：`![描述](../images/文章主题/图片名.jpg)`

**5. 验证图片显示**

**配图原则：**
- 图片要清晰（宽度至少1000px）
- 图片要相关（不用无关的装饰图）
- 注明来源（如需）

## 7个关键要点（贯穿整个流程）

1. **Think Aloud透明化思考** - 每次决策都说明思考过程
2. **调研先行** - 新概念、新技术必须搜索验证
3. **选题讨论必做** - 不要直接写，先讨论方向
4. **个人素材库降AI味** - 用真实经历替代AI腔
5. **三遍审校机制** - 系统化降低AI味
6. **文章配图流程** - 直接完成，不只是写指南
7. **协作文档** - 明确分工，让用户知道需要配合什么

---

## 流程灵活性说明

**流程是指南，不是教条；核心原则不可妥协。**

**流程可以灵活调整：**
- 用户明确要求跳过某步骤 → 可以遵循（但提醒风险）
- 任务特别简单/紧急 → 可以简化流程
- 上下文已包含所需信息 → 不重复操作

**但核心原则不能妥协：**
- ❌ 绝不编造数据
- ❌ 绝不使用过时信息
- ❌ 绝不省略Think Aloud
- ❌ 绝不跳过用户确认（重要决策）
