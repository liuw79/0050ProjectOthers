# macOS 系统故障排查与维护

本文档记录 macOS 系统遇到的问题及其解决方案。

## iMac 未能正常进入低功耗待机状态

**问题诊断日期**: 2025-06-28

### 现象

iMac 在过去24小时内未能按预期进入低功耗的待机或睡眠模式，即使在用户未操作的情况下也持续处于工作状态。

### 诊断过程

通过 `pmset -g log` 和 `pmset -g assertions` 命令对系统电源管理日志进行分析，发现以下几个进程是阻止系统进入睡眠的主要原因：

1.  **时间机器备份 (`backupd`)**: 系统日志显示，名为 `Backup Job` 的进程持有一个 `PreventUserIdleSystemSleep` 断言。这意味着时间机器正在执行备份任务，为了防止数据损坏，它会阻止系统进入睡眠，直到备份完成。

2.  **Spotlight 索引 (`mds`, `mdsync`)**: 名为 `com.apple.metadata.mds.scan` 的进程持有一个 `BackgroundTask` 断言。这表明 Spotlight 正在为磁盘文件创建索引以便于快速搜索。此过程（尤其是在添加大量新文件或连接新硬盘后）可能会消耗较长时间并阻止系统休眠。

### 结论

iMac 未能进入待机模式并非故障，而是系统正在后台执行重要的维护任务（数据备份和文件索引）。这些任务的优先级较高，会暂时阻止系统进入睡眠以确保任务顺利完成。

根据日志分析，在过去的一天中，您的 iMac **基本没有进入过待机状态**，绝大部分时间都因为上述后台任务而保持工作。当这些任务完成后，系统应该能恢复正常的睡眠行为。

### 建议

- **耐心等待**: 通常情况下，让这些后台任务自行完成是最好的选择。特别是首次备份或索引，耗时会较长。
- **检查任务状态**: 您可以随时通过以下方式检查任务进度：
    - **时间机器**: 打开“系统设置” -> “时间机器”，查看备份状态。
    - **Spotlight**: 点击菜单栏的 Spotlight 搜索图标（放大镜），如果正在索引，通常会显示一个进度条。
- **手动暂停**: 如果您需要立即让 Mac 睡眠，可以手动暂停这些任务，但这可能会影响备份的完整性和搜索的准确性。

### 跟进诊断：发现 `coreaudiod` 成为新的睡眠阻止者

在解决了 `bluetoothd` 的问题后，通过持续监控发现，音频服务 `coreaudiod` 有时也会阻止iMac进入睡眠状态，导致持续的高能耗。

#### 解决方案 3.0：服务的“按需启停”

为了更优雅地解决这个问题，我们采用了一套“按需启停”的方案，而不是粗暴地重启服务。

**原理**：

1.  **休眠前卸载**：当监控脚本检测到 `coreaudiod` 正在阻止睡眠时，它会使用 `launchctl unload` 命令来“卸载”该服务。这会使服务停止，从而允许iMac顺利进入休眠。
2.  **唤醒后重载**：我们创建了一个由 `launchd` 管理的唤醒代理。当iMac从睡眠中被唤醒时，这个代理会自动触发一个脚本，使用 `launchctl load` 命令重新加载 `coreaudiod` 服务，确保音频功能恢复正常。

**优势**：

*   **精确节能**：只在需要休眠时才停止服务，避免了不必要的服务中断和重启。
*   **无缝体验**：用户唤醒iMac后，音频服务会自动恢复，不影响正常使用。
*   **高可扩展性**：这套框架可以轻松扩展，用于管理其他可能阻止睡眠的进程。

### `DictationIM` 进程崩溃分析

根据用户提供的崩溃报告 (`2025-07-02`)，我们对 `DictationIM` (语音听写输入法) 进程的崩溃进行了分析。

*   **进程信息**:
    *   **进程名**: `DictationIM`
    *   **路径**: `/System/Library/Input Methods/DictationIM.app`
    *   **异常类型**: `EXC_BAD_INSTRUCTION (SIGILL)` - 非法指令，通常意味着程序试图执行一个CPU无法识别的指令。
*   **崩溃线程**: `Thread 14 - com.apple.speech.RXAPIDispatchQueue`
*   **关键错误信息**: `BUG IN CLIENT OF LIBDISPATCH: dispatch_sync called on queue already owned by current thread`

### DictationIM 是什么？

`DictationIM` 是 macOS 系统中负责**语音听写**（Dictation）功能的后台进程。它作为一个输入法（Input Method, IM）运行，捕捉您的语音并将其转换为文本。

当您启用“听写”功能时，`DictationIM` 会在需要时启动，并请求`coreaudiod`服务来访问麦克风。正常情况下，当听写结束或您切换到其他输入法时，`DictationIM` 会通知 `coreaudiod` 释放音频资源。然而，如崩溃报告所示，`DictationIM` 内部的线程死锁导致其异常终止，这个“通知”过程未能完成，从而留下了无法自动清除的“僵尸”断言。

### 结论与最终解决方案

`DictationIM` 进程的崩溃是导致 `coreaudiod` 持久化睡眠阻止断言的根本原因。该进程在异常终止时未能正确释放其占用的音频资源，导致系统睡眠失败。

在尝试了多种复杂的脚本监控和进程管理方案后，最终发现最简单且最有效的解决方案是直接从系统层面禁用相关功能。

#### 首选解决方案：关闭语音控制

实践证明，最根本的解决方法是**彻底禁用语音控制（Voice Control）功能**。这会阻止 `DictationIM` 和 `com.apple.SpeechRecognitionCore` 等相关进程的启动，从而根除其导致的 `coreaudiod` 睡眠断言问题。

**操作步骤：**

1.  打开 **系统设置** (System Settings)。
2.  进入 **辅助功能** (Accessibility)。
3.  在左侧列表中选择 **语音控制** (Voice Control)。
4.  **关闭** “语音控制”功能的开关。

**此方法被验证为最终解决方案，简单直接，且无任何副作用。**

#### 备选方案：禁用语音听写（若上述方法无效）

如果仅仅关闭语音控制后问题依旧存在，可以进一步禁用语音听写功能。

1.  打开 **系统设置** (System Settings)。
2.  进入 **键盘** (Keyboard) 设置。
3.  向下滚动找到 **听写** (Dictation) 部分。
4.  **关闭** “听写”功能的开关。
5.  系统可能会询问您是否确定，请确认关闭。

**执行此操作后，`DictationIM` 进程将不会再被激活，从而解决了导致睡眠失败的问题。**

**具体实现**：

*   **监控脚本 (`imac_sleep_monitor.sh`)**：增加了检测 `coreaudiod` 并使用 `launchctl unload` 的逻辑。
*   **唤醒脚本 (`wake_up_services.sh`)**：一个新创建的脚本，专门用于在唤醒时 `launchctl load` 服务。
*   **Launchd 代理 (`com.user.wake_up_services.plist`)**：一个 `.plist` 文件，定义了在系统唤醒时触发唤醒脚本的规则。

这套组合方案最终实现了对iMac睡眠问题的智能化、自动化管理。

### 跟进诊断：`bluetoothd` 成为新的睡眠阻止者

**情况更新**

用户反馈时间机器的备份目标NAS并未开机，因此持续的备份尝试非常不合理。在关闭时间机器后，再次运行 `pmset -g assertions` 进行检查。

**新发现**

1.  **时间机器影响已消除**: `backupd` 进程持有的 `PreventUserIdleSystemSleep` 断言消失。这证实了**时间机器在无法连接到备份磁盘时，是阻止系统睡眠的主要原因**。

2.  **新的睡眠阻止者 - 蓝牙服务**: 日志显示，新的 `PreventUserIdleSystemSleep` 断言由以下进程持有：
    *   `bluetoothd`: 系统的核心蓝牙服务。
    *   `useractivityd`: 与蓝牙低功耗设备（BTLE）广播相关的服务。

**结论更新**

关闭时间机器后，系统无法休眠的根本原因转移到了**蓝牙活动**上。这可能是由连接的蓝牙外设（如鼠标、键盘）或系统功能（如Handoff）引起的。

**新建议**

如果问题持续，可以尝试以下步骤排查蓝牙问题：

- **检查蓝牙设备**: 断开所有非必要的蓝牙设备，观察iMac是否能正常睡眠。
- **关闭“蓝牙唤醒”功能**: 在“系统设置” -> “蓝牙” -> “高级”中，尝试关闭“允许蓝牙设备唤醒此电脑”选项。
- **关闭Handoff**: 在“系统设置” -> “通用” -> “AirDrop与接力”中，关闭“允许在这台Mac和iCloud设备之间使用接力功能”。

### 自动化解决方案与迭代

为了彻底解决 `bluetoothd` 导致的问题，我们尝试了自动化脚本进行干预。

#### 初版方案：定时关闭蓝牙 (存在缺陷)

*   **方案**: 创建一个监控脚本，通过 `cron` 定时任务每15分钟检查一次睡眠阻止者。如果发现是 `bluetoothd`，则使用 `blueutil` 工具强行关闭蓝牙。同时，创建一个 `launchd` 代理，在系统唤醒时自动重新开启蓝牙。
*   **缺陷**: 正如用户指出的，此方案的致命缺陷在于，**关闭蓝牙后，将无法使用蓝牙键盘或鼠标来唤醒 iMac**。这严重影响了用户的核心使用体验，因此该方案不可取。

#### 最终方案：重启蓝牙服务 (当前采用)

*   **原理**: 经过进一步分析，问题根源在于 `bluetoothd` 服务进程本身可能进入了某个不稳定的状态，而不是蓝牙射频模块本身的问题。因此，我们不需要关闭整个蓝牙功能，只需重启服务进程即可。
*   **方案**: 修改监控脚本，当检测到 `bluetoothd` 阻止睡眠时，不再使用 `blueutil` 关闭蓝牙，而是通过 `launchctl` 命令来优雅地重启 `bluetoothd` 服务。具体命令为 `launchctl kickstart -k system/com.apple.bluetoothd`。
*   **优势**: 
    1.  **不影响唤醒**: 该操作仅重启服务进程，蓝牙射频模块一直处于开启状态，因此不影响通过蓝牙设备唤醒电脑。
    2.  **解决问题**: 重启服务可以有效地将其从异常状态中恢复，从而解除其对系统睡眠的阻止。
    3.  **简化部署**: 由于不再需要“唤醒后开启蓝牙”这个步骤，我们也就不再需要 `launchd` 代理，整个方案仅依赖一个 `cron` 定时任务，部署和维护都更加简单。
