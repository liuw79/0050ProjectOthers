# GTM产品市场协同作战在线表格系统设计文档

## 1. 项目概述

### 1.1 项目背景

GTM（Go-To-Market，产品上市）是连接IPD（集成产品开发）和IPMS（集成产品管理系统）的关键桥梁，对于产品成功上市至关重要。本项目旨在开发一个在线表格系统，支持多用户协同编辑GTM工具表，提高团队协作效率和产品上市成功率。

### 1.2 项目目标

开发一个简单易用、功能完备的在线表格系统，支持：

- 多用户实时协同编辑GTM工具表
- 表格数据的实时保存和同步
- 简洁直观的用户界面
- 稳定可靠的系统性能

### 1.3 MVP范围

MVP（最小可行产品）阶段将专注于实现核心功能：

- 表格的基本编辑功能
- 多用户实时协作
- 基本用户管理

暂不实现的功能：

- 复杂的权限管理系统
- 高级数据分析功能
- 自定义工作流

## 2. 系统架构

### 2.1 总体架构

系统采用前后端分离的架构设计，包括：

- **前端**：基于React的单页面应用
- **后端**：Node.js API服务
- **数据库**：MongoDB文档数据库
- **实时通信**：WebSocket服务

### 2.2 前端架构

前端采用现代化的React技术栈：

- **框架**：React 18
- **状态管理**：Redux Toolkit
- **UI组件库**：Ant Design
- **表格组件**：Handsontable或AG Grid
- **实时通信**：Socket.io客户端
- **构建工具**：Vite

### 2.3 后端架构

后端采用轻量级的Node.js技术栈：

- **运行环境**：Node.js
- **Web框架**：Express.js
- **API规范**：RESTful API
- **实时通信**：Socket.io服务端
- **数据验证**：Joi或Yup

### 2.4 部署架构

MVP阶段采用简单的部署方案：

- **前端**：静态文件部署在CDN或云存储
- **后端**：容器化部署在云服务器
- **数据库**：MongoDB Atlas云数据库服务

## 3. 数据库设计

### 3.1 数据库选型

选择MongoDB作为数据库，原因如下：

- 灵活的文档模型，适合存储结构可能变化的表格数据
- 良好的水平扩展能力
- 支持复杂查询和索引
- 与Node.js生态系统集成良好

### 3.2 核心数据模型

#### 3.2.1 用户模型（User）

```javascript
{
  _id: ObjectId,
  username: String,  // 用户名
  email: String,     // 电子邮件
  password: String,  // 加密密码
  displayName: String, // 显示名称
  avatar: String,    // 头像URL
  createdAt: Date,   // 创建时间
  updatedAt: Date    // 更新时间
}
```

#### 3.2.2 表格模型（Sheet）

```javascript
{
  _id: ObjectId,
  name: String,      // 表格名称
  description: String, // 表格描述
  createdBy: ObjectId, // 创建者ID
  createdAt: Date,   // 创建时间
  updatedAt: Date,   // 更新时间
  collaborators: [   // 协作者列表
    {
      userId: ObjectId, // 用户ID
      accessTime: Date  // 最近访问时间
    }
  ]
}
```

#### 3.2.3 表格数据模型（SheetData）

```javascript
{
  _id: ObjectId,
  sheetId: ObjectId, // 关联的表格ID
  data: Object,      // 表格数据（JSON格式）
  version: Number,   // 数据版本号
  updatedAt: Date,   // 更新时间
  updatedBy: ObjectId // 更新者ID
}
```

#### 3.2.4 操作历史模型（OperationHistory）

```javascript
{
  _id: ObjectId,
  sheetId: ObjectId, // 关联的表格ID
  userId: ObjectId,  // 操作用户ID
  operation: Object, // 操作详情
  timestamp: Date    // 操作时间
}
```

## 4. 实时协作功能

### 4.1 技术实现

实时协作功能基于WebSocket技术实现，使用Socket.io库：

- **客户端连接管理**：处理用户连接、断开和重连
- **操作广播**：将用户操作实时广播给其他协作者
- **冲突解决**：基于操作转换（OT）或最终一致性策略
- **状态同步**：定期同步表格完整状态，确保数据一致性

### 4.2 核心功能

#### 4.2.1 实时编辑

- 用户编辑单元格时，变更实时传输给所有协作者
- 显示其他用户的光标位置和选择区域
- 编辑冲突时采用后到者优先策略

#### 4.2.2 协作者状态

- 显示当前在线协作者列表
- 标识每个协作者的编辑区域
- 提供简单的协作者通信机制

#### 4.2.3 变更历史

- 记录所有编辑操作
- 支持查看历史变更
- 提供简单的撤销/重做功能

## 5. API接口设计

### 5.1 用户管理API

| 接口 | 方法 | 描述 |
|------|------|------|
| `/api/auth/register` | POST | 用户注册 |
| `/api/auth/login` | POST | 用户登录 |
| `/api/auth/logout` | POST | 用户登出 |
| `/api/users/me` | GET | 获取当前用户信息 |
| `/api/users/:id` | GET | 获取指定用户信息 |

### 5.2 表格管理API

| 接口 | 方法 | 描述 |
|------|------|------|
| `/api/sheets` | GET | 获取用户可访问的表格列表 |
| `/api/sheets` | POST | 创建新表格 |
| `/api/sheets/:id` | GET | 获取表格详情 |
| `/api/sheets/:id` | PUT | 更新表格信息 |
| `/api/sheets/:id` | DELETE | 删除表格 |

### 5.3 表格数据API

| 接口 | 方法 | 描述 |
|------|------|------|
| `/api/sheets/:id/data` | GET | 获取表格数据 |
| `/api/sheets/:id/data` | PUT | 更新表格数据 |
| `/api/sheets/:id/history` | GET | 获取表格操作历史 |

### 5.4 WebSocket事件

| 事件 | 方向 | 描述 |
|------|------|------|
| `join_sheet` | 客户端→服务端 | 加入表格协作 |
| `leave_sheet` | 客户端→服务端 | 离开表格协作 |
| `cell_update` | 双向 | 单元格更新 |
| `selection_change` | 双向 | 选择区域变化 |
| `collaborator_joined` | 服务端→客户端 | 协作者加入 |
| `collaborator_left` | 服务端→客户端 | 协作者离开 |

## 9. 技术选型理由

### 9.1 前端技术

- **React**：组件化开发，丰富的生态系统，良好的性能
- **Redux Toolkit**：简化状态管理，提供标准化的状态更新模式
- **Ant Design**：成熟的企业级UI组件库，美观且功能丰富
- **Handsontable/AG Grid**：专业的表格组件，支持复杂的表格操作

### 9.2 后端技术

- **Node.js/Express**：轻量级、高性能，适合构建API服务
- **Socket.io**：成熟的WebSocket库，支持实时双向通信
- **MongoDB**：灵活的文档数据库，适合存储结构可变的表格数据

### 9.3 部署技术

- **Docker**：容器化部署，环境一致性
- **云服务**：弹性扩展，按需付费
