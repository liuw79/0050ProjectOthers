# 会议纪要模板参考

## 标准会议纪要格式

### 基本信息模板

```markdown
# [会议标题]

**日期**: YYYY-MM-DD
**时间**: HH:MM - HH:MM
**地点**: [会议室/线上链接]
**参会人员**: 张三, 李四, 王五
**记录人**: 赵六

## 会议议程

1. 议题一
2. 议题二
3. 议题三

## 讨论内容

### 议题一: [标题]

**讨论要点**:
- 要点1
- 要点2

**决策**: [具体决策内容]

### 议题二: [标题]

**讨论要点**:
- 要点1
- 要点2

**决策**: [具体决策内容]

## 待办任务

- [ ] 任务1 - @负责人 ⏰ 截止日期
- [ ] 任务2 - @负责人 ⏰ 截止日期
- [ ] 任务3 - @负责人 ⏰ 截止日期

## 下次会议

**时间**: YYYY-MM-DD HH:MM
**议程**:
- 议题1
- 议题2
```

## 关键词识别指南

### 决策关键词

系统会自动识别以下关键词来提取决策:
- 决定、决策、批准、同意
- 确定、定为、决议
- 通过、采纳

**示例**:
- "会议决定采用方案A进行开发"
- "批准预算申请,金额为50万元"

### 任务关键词

系统会识别任务标记和关键词:

**任务标记**:
- `- [ ]` 或 `* [ ]` (Markdown复选框)
- `□` 或 `☐` (空心方框)
- `1.` `2.` (编号列表)

**任务关键词**:
- 任务、待办、需要
- 负责、完成、交付
- TODO、Action Item

**示例**:
- "- [ ] 完成需求文档编写 @张三 ⏰ 本周五"
- "任务1: 李四负责数据分析报告,截止下周一"

### 建议关键词

系统会识别下次会议相关的内容:
- 建议、提议、下次
- 后续、跟进
- 下周、下一步

**示例**:
- "建议下次会议邀请技术团队参加"
- "下一步需要准备竞品分析材料"

## 最佳实践

### 1. 清晰的任务分配

✅ 好的示例:
```
- [ ] 完成用户调研报告 @王五 ⏰ 2025-02-05
- [ ] 设计原型图 @李四 ⏰ 下周三
```

❌ 避免:
```
- 需要有人做调研
- 原型图要尽快完成
```

### 2. 明确的决策记录

✅ 好的示例:
```
**决策**: 采用微服务架构,预算控制在100万元以内,由技术团队负责实施
```

❌ 避免:
```
大家讨论了一下架构问题,可能会用微服务
```

### 3. 结构化的内容组织

使用标题、列表、分段来组织内容,便于解析:

```markdown
## 技术方案讨论

### 方案对比
- 方案A: 优点...缺点...
- 方案B: 优点...缺点...

### 最终决策
决定采用方案A

## 后续行动
- [ ] 技术选型 @张三
- [ ] 环境搭建 @李四
```

## 导入格式支持

系统支持以下格式的会议纪要:

1. **Markdown文件** (.md) - 推荐
2. **纯文本文件** (.txt)
3. **Word文档** (.docx)
4. **PDF文件** (.pdf)

### Markdown格式优势

- 结构清晰,易于解析
- 支持复选框、编号列表
- 可直接在GitHub/Notion等平台查看
- 便于版本控制

## 输出格式说明

### JSON格式

包含完整的结构化数据,适合程序处理:

```json
{
  "meeting_info": {...},
  "decisions": [...],
  "tasks": [...],
  "next_meeting_suggestions": [...]
}
```

### Markdown任务清单

人类可读的任务清单,包含:
- 会议信息汇总
- 决策事项列表(带优先级)
- 待办任务清单(带负责人和截止日期)
- 下次会议建议

### 项目看板

按照看板方式组织任务:
- 待办列
- 进行中列
- 已完成列
- 里程碑
- 下一步行动

### 甘特图数据

Mermaid格式的甘特图,可直接在Markdown中渲染或导入项目管理工具。
